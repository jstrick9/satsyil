8
Two Intake Channels
IFILE Web Portal: browser-based submission UI for direct filers; WCAG 2.2 AA; Login.gov authentication at IAL-1/IAL-2
Bulk Filing API: RESTful JSON-first with backward-compatible XML for third-party software providers; API key authentication
All submissions enter a controlled SQS intake layer — decoupled from processing, durable, and loss-proof even under peak load
Unique tracking ID issued immediately upon intake — filer receives acknowledgment before processing begins
Pre-validation executes before formal acceptance: schema check, field-level rules, entity validation (EIN, plan number)
AWS Step Functions orchestrate the processing pipeline: validate → process → distribute — exactly-once, auditable, stateless
Validated filings distributed via encrypted SFTP/API to DOL, IRS, and PBGC per the IRD; NARA annual archive packages prepared in non-proprietary formats with full metadata
Public data dissemination via OpenSearch-powered search portal — processed filings made accessible per disclosure requirements
Processing Pipeline (5 Stages)
① Intake: Submission received via IFILE or Bulk API; SQS queued; tracking ID issued; acknowledgment sent
② Validate: Schema validated against DER; field-level, cross-field, and entity rules applied; errors returned with actionable guidance
③ Process: Step Functions execute processing workflow; exactly-once delivery enforced via idempotency keys
④ Acknowledge: Real-time status updates delivered to filer via API and dashboard at each milestone
⑤ Distribute: Encrypted delivery to DOL, IRS, PBGC drop-boxes; NARA archive; OpenSearch public portal updated
All data in motion: TLS 1.3 + FIPS-validated cryptography
All data at rest: AWS KMS customer-managed keys
Every stage logged with timestamp and audit record — full DOL visibility into pipeline at any moment

Queuing & Reliability Controls
Amazon SQS intake queue: every submission durably queued the moment it is received — no filing is lost even if all downstream services are temporarily unavailable
Exactly-once delivery: SQS + Step Functions use idempotency keys — each filing is processed once and only once; no duplicates, no skipped filings
Automated retries with exponential backoff: transient failures (network blips, DB timeouts) self-heal automatically without any human intervention
Dead-letter queues (DLQs): after configurable retry exhaustion, failed filings routed to DLQ; ops team alerted within 60 seconds with filing ID, failure reason, and processing stage
Pre-submission validation: malformed or non-compliant submissions rejected before entering the processing queue — dramatically reduces exception volume at source
Duplicate detection: filing fingerprint hashing prevents double-processing of the same submission; idempotent resubmission supported
End-to-end traceability: every filing has a unique tracking ID from intake through distribution; every status transition logged with timestamp and audit record
DOL visibility: shared CloudWatch dashboard provides real-time view of queue depth, processing throughput, error rates, and DLQ volume at any moment
Exception Handling Workflow
Exception triage: DLQ alerts include structured metadata — filing ID, submitter, form type, failure code, processing stage — enabling rapid root cause analysis
Automated remediation: common exception classes (schema version mismatch, temporary downstream outage) trigger automated resolution workflows without ops team involvement
Manual remediation: ops team reviews DLQ entries; corrective action documented in Jira with DOL visibility; filing reprocessed once issue resolved
Filer notification: if a filing cannot be processed after all retries, the filer receives a structured error notification with specific guidance for resubmission
Quality control gates: automated validation checks at each pipeline stage prevent downstream propagation of bad data
Operational metrics reviewed weekly: exception rates, DLQ volume, retry frequency, and resolution time tracked against SLA targets
Continuous improvement: recurring exception patterns trigger root cause analysis and permanent system fix — not just one-time remediation
Zero silent failures: every filing that enters the system either reaches a successful distribution state or generates a tracked, alerted, documented exception — no filing disappears

Queue Management
Amazon SQS decouples intake from processing: every submission durably stored in queue before processing begins — no data loss even if processing is temporarily unavailable
Exactly-once delivery: SQS and AWS Step Functions use idempotency keys — each filing processed once and only once; no duplicates regardless of retry count
Queue depth monitoring: CloudWatch tracks queue depth per provider and overall; auto-scaling triggers when depth exceeds threshold
FIFO ordering: within each provider's submission batch, filings processed in submission order where order matters
Dead-letter queue (DLQ): filings that exhaust all retries move to DLQ; ops team alerted within 60 seconds with filing ID, provider, and failure context
DLQ monitoring: DLQ depth tracked separately; any DLQ entries require investigation and resolution — no silent discard
Priority queuing: DOL can configure priority lanes for specific submission types or peak periods
Queue visibility: Satsyil and DOL can view real-time queue state; queue metrics included in monthly operational report
Retry Logic & Exception Handling
Automated retry with exponential backoff: transient failures retried automatically; backoff prevents thundering herd during recovery from outage
Retry limits: configurable per error class; transient errors retry up to 5 times; permanent errors (validation failures) do not retry
Retry visibility: each retry logged with attempt number, delay, and outcome; full retry history available for any filing
Exception classification: transient (network/timeout), permanent (validation failure), or system (infrastructure) — different handling for each class
System exception escalation: infrastructure-level exceptions trigger PagerDuty alert to ops team; resolved within SLA regardless of time of day
Provider notification on permanent failure: when a filing exhausts retries, provider receives structured failure notice with all applicable error codes
Exception trending: exception rates by type tracked weekly; increases trigger root cause investigation before they become incidents
Recovery SLA: no filing left in DLQ for more than 24 hours without resolution or explicit DOL-approved exception
